# 자기소개서

## 지원 동기

저는 15년 이상 실시간 서비스 환경에서 클라이언트부터 서버, 데이터베이스까지 풀스택으로 개발해왔습니다. 기능 구현 자체보다 **"운영 환경에서 정확하게, 예측 가능하게 동작하는가"**를 가장 중요한 기준으로 삼아 왔습니다. 동시접속자가 많고 데이터가 빠르게 오가는 시스템에서는 작은 정합성 오류가 곧 장애로 이어집니다. 그래서 저는 "일단 되면 된다"가 아니라, 정확성·정합성·재현성을 확보하는 방향으로 설계하고 운영해왔습니다.

금융 시스템은 그 기준이 더욱 엄격한 도메인이라고 이해하고 있습니다. 수치의 오차, 처리의 누락/중복, 상태 불일치가 곧 고객 신뢰와 리스크로 이어지기 때문에, 제가 익혀온 실서비스 운영 경험과 검증 중심의 개발 습관이 가장 의미 있게 쓰일 수 있는 분야라고 판단하여 지원하게 되었습니다.

---

## 제가 해온 일들

### 대규모 수치 연산 시스템

실시간 서비스에서 150개가 넘는 수치가 동시에 계산되는 시스템을 만든 적이 있습니다. 여러 파라미터가 서로 영향을 주는 구조라서, 계산 순서나 타이밍이 조금만 어긋나도 결과가 달라집니다. 캐싱이랑 이벤트 기반 갱신 구조를 써서, 어떤 타이밍에 조회해도 정확한 값이 나오도록 설계했습니다.

### 실시간 데이터 처리

클라이언트-서버 간 실시간 동기화가 필수인 환경에서 여러 프로젝트를 했습니다:
- gRPC, WebSocket 기반 저지연 통신 구조 설계
- 네트워크 끊겼을 때 재시도 로직, 데이터 복구 처리
- 서버에서 검증하는 부정 행위 탐지 로직 구현

### 결제 시스템

Google Play, App Store, OneStore 결제를 전부 통합해서 처리하는 시스템을 만들었습니다. 영수증 검증이랑 정산 로직도 직접 짰습니다. 돈이 오가는 부분이라 특히 신경 많이 썼습니다.

### 데이터베이스 설계 및 API 개발

PHP + MySQL 기반으로 서버 데이터 구조랑 API를 직접 설계한 경험이 있습니다. 테이블 설계부터 쿼리 최적화까지 했고, 클라이언트-서버 간 데이터 흐름이 꼬이지 않도록 구조를 잡았습니다.

### 대규모 서비스 운영

수십만~백만 명이 쓰는 서비스를 운영해봤습니다. 메모리 누수 잡고, 크래시 원인 분석하고, 무중단으로 업데이트 배포하는 것까지 직접 했습니다. 몇 년 된 레거시 코드 정리하면서 "돌아가는 서비스는 함부로 건드리면 안 된다"는 것도 배웠습니다.

## 일하는 방식

요구사항 파악하는 데 시간을 많이 씁니다. "이거 왜 필요해요?"를 먼저 물어보고, 의도를 이해한 다음에 코드를 짜는 편입니다. 설계 단계에서 테이블 구조 리뷰에 참여하고, 화면부터 서버까지 맡은 영역은 책임지고 끝까지 개발합니다. 결과물이 제대로 동작하는지가 가장 중요하다고 생각합니다.

## 마무리

게임 쪽에서 15년 일하면서, 다양한 예외 상황을 미리 고려하는 습관이 생겼습니다. 유저들은 개발자가 생각하지 못한 방식으로 시스템을 사용하고, 예상 밖의 입력값이 들어오는 경우도 많습니다. 그래서 "이런 경우엔 어떻게 되지?"를 항상 먼저 생각하게 됐습니다.

금융 쪽은 회사마다 쓰는 툴이나 업무 방식이 다르다고 들었습니다. 새로운 환경이지만, 맡은 업무를 정확히 파악하고 그에 맞춰 개발하는 건 어디서든 같다고 생각합니다. 팀원들과 소통이 원활하면 일도 수월해지고, 결국 중요한 건 개발 결과물에 대한 신뢰라고 생각합니다.
