# 자기소개서

## 지원 동기

15년 넘게 소프트웨어를 만들어오면서, 제가 가장 신경 쓰는 부분은 항상 "이게 진짜 맞게 돌아가는가"였습니다. 금융 쪽은 숫자 하나 틀리면 큰일 난다고 들었고, 그게 제가 일해온 방식과 맞닿아 있다고 느꼈습니다.

저는 주로 실시간으로 돌아가는 서비스를 만들어왔습니다. 동시접속자가 많은 환경에서 서버랑 클라이언트 사이에 데이터가 어긋나면 바로 문제가 터지거든요. 그래서 "일단 되면 됐지" 하고 넘어간 적이 없습니다. 이런 경험을 금융 시스템에서 써먹을 수 있을 것 같아서 지원하게 됐습니다.

---

## 제가 해온 일들

### 대규모 수치 연산 시스템

실시간 서비스에서 150개가 넘는 수치가 동시에 계산되는 시스템을 만든 적이 있습니다. 여러 파라미터가 서로 영향을 주는 구조라서, 계산 순서나 타이밍이 조금만 어긋나도 결과가 달라집니다. 캐싱이랑 이벤트 기반 갱신 구조를 써서, 어떤 타이밍에 조회해도 정확한 값이 나오도록 설계했습니다.

### 실시간 데이터 처리

클라이언트-서버 간 실시간 동기화가 필수인 환경에서 여러 프로젝트를 했습니다:
- gRPC, WebSocket 기반 저지연 통신 구조 설계
- 네트워크 끊겼을 때 재시도 로직, 데이터 복구 처리
- 서버에서 검증하는 부정 행위 탐지 로직 구현

### 결제 시스템

Google Play, App Store, OneStore 결제를 전부 통합해서 처리하는 시스템을 만들었습니다. 영수증 검증이랑 정산 로직도 직접 짰습니다. 돈이 오가는 부분이라 특히 신경 많이 썼습니다.

### 대규모 서비스 운영

수십만~백만 명이 쓰는 서비스를 운영해봤습니다. 메모리 누수 잡고, 크래시 원인 분석하고, 무중단으로 업데이트 배포하는 것까지 직접 했습니다. 몇 년 된 레거시 코드 정리하면서 "돌아가는 서비스는 함부로 건드리면 안 된다"는 것도 배웠습니다.

## 일하는 방식

요구사항 파악하는 데 시간을 많이 씁니다. "이거 왜 필요해요?"를 먼저 물어보고, 의도를 이해한 다음에 코드를 짜는 편입니다. 서버 개발자랑 데이터 구조 맞추는 것도 중요하게 생각합니다.

금융 쪽은 코드 리뷰나 감사 대응이 중요하다고 들었는데, 저도 원래 제 코드는 남이 봐도 이해할 수 있게 짜려고 노력합니다. 나중에 제가 없어도 유지보수 되어야 하니까요.
